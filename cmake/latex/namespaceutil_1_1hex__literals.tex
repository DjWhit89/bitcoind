\doxysubsection{Detailed Description}
"{}"{}\+\_\+hex is a compile-\/time user-\/defined literal returning a {\ttfamily std\+::array$<$std\+::byte$>$}, equivalent to \doxyref{Parse\+Hex()}{p.}{strencodings_8h_a882aa30dcd768538f02d7e710bf44dc9}. Variants provided\+:


\begin{DoxyItemize}
\item "{}"{}\+\_\+hex\+\_\+v\+: Returns {\ttfamily std\+::vector$<$std\+::byte$>$}, useful for heap allocation or variable-\/length serialization.
\item "{}"{}\+\_\+hex\+\_\+u8\+: Returns {\ttfamily std\+::array$<$uint8\+\_\+t$>$}, for cases where {\ttfamily std\+::byte} is incompatible.
\item "{}"{}\+\_\+hex\+\_\+v\+\_\+u8\+: Returns {\ttfamily std\+::vector$<$uint8\+\_\+t$>$}, combining heap allocation with {\ttfamily uint8\+\_\+t}.
\end{DoxyItemize}

\begin{DoxyWarning}{Warning}
It could be necessary to use vector instead of array variants when serializing, or vice versa, because vectors are assumed to be variable-\/ length and serialized with a size prefix, while arrays are considered fixed length and serialized with no prefix.

It may be preferable to use vector variants to save stack space when declaring local variables if hex strings are large. Alternatively variables could be declared constexpr to avoid using stack space.

Avoid {\ttfamily uint8\+\_\+t} variants when not necessary, as the codebase migrates to use {\ttfamily std\+::byte} instead of {\ttfamily unsigned char} and {\ttfamily uint8\+\_\+t}.
\end{DoxyWarning}
\begin{DoxyNote}{Note}
One reason "{}"{}\+\_\+hex uses {\ttfamily std\+::array} instead of {\ttfamily std\+::vector} like \doxyref{Parse\+Hex()}{p.}{strencodings_8h_a882aa30dcd768538f02d7e710bf44dc9} does is because heap-\/based containers cannot cross the compile-\/ time/runtime barrier. 
\end{DoxyNote}
